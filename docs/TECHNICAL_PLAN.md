# TECHNICAL_PLAN — детальный технический план `aiBrowserPlagin`

> Технический компаньон к [`PLAN.md`](./PLAN.md). Что вне MVP —
> в [`POST_MVP_PLAN.md`](./POST_MVP_PLAN.md).
>
> Терминология этапов — **«Этап N»** (как в соседних проектах файловой системы,
> напр. `ai-knowledge-base`).

---

## 1. Вводные и ограничения

- Соло-разработчик-интроверт, реализация с AI-ассистентом; приоритет —
  **тренировка кодирования** (TypeScript + Manifest V3 + Vite).
- Экономическая обоснованность **не обязательна**; целевые расходы автора
  ≈ **$0/мес** — расширение полностью клиентское, бэкенда нет.
- **BYOK**: ключ к LLM приносит пользователь; ключ — только в `chrome.storage`,
  никогда не в коде/git/логах.
- Узкий скоуп — не Shelf (см. `PLAN.md §3`): без БД, эмбеддингов, RAG, графа.
- Целевая платформа MVP — **Chrome/Chromium** (Manifest V3). Firefox — post-MVP.

## 2. Технический стек

| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | TypeScript 5.x (strict) | Естественный для расширений, типобезопасность message-passing |
| Платформа | Manifest V3 | Единственный поддерживаемый формат в актуальном Chrome |
| Сборщик | **Vite + `@crxjs/vite-plugin`** | См. ниже |
| UI popup | голый TS + минимальный HTML/CSS | Крошечная поверхность, React избыточен на MVP (см. §5) |
| LLM-доступ | `fetch` (stdlib браузера) из service worker | Без SDK — прямой OpenAI-совместимый вызов |
| PDF | `pdf.js` (`pdfjs-dist`) | Индустриальный стандарт извлечения текста из PDF |
| Хранилище настроек | `chrome.storage.local` | Штатный runtime-storage браузера для BYOK |
| Тесты | Vitest (юнит, добавляется с Этапа 1) | Лёгкий, нативно дружит с Vite |

### Почему `@crxjs/vite-plugin`, а не голый Vite / webpack / wxt

- **Знает про Manifest V3.** Читает типизированный `manifest.config.ts`, сам
  бандлит service worker / content scripts / popup HTML как отдельные entry,
  переписывает пути ассетов, генерирует корректный `dist/manifest.json`. Без
  плагина всё это пришлось бы описывать вручную в `build.rollupOptions`.
- **HMR для расширения** в `vite dev` — content scripts и popup перезагружаются
  без ручной пересборки; для одиночной разработки это заметно ускоряет цикл.
- **Vite-экосистема** — быстрый esbuild-транспайл, привычный конфиг, лёгкая
  интеграция Vitest.
- Альтернативы отклонены: голый webpack — больше бойлерплейта под MV3; `wxt` —
  более «фреймворковый», прячет манифест за своими абстракциями (меньше учебной
  ценности для понимания MV3 напрямую), и тяжелее для узкого скоупа.

Осознанное ограничение зависимостей: LLM-вызов — на голом `fetch` (тренировка
ручного HTTP-клиента, retry, обработки ошибок), без официального SDK.

## 3. Структура проекта

```
aiBrowserPlagin/
├── manifest.config.ts        # типизированный Manifest V3 (defineManifest), источник версии/путей
├── vite.config.ts            # Vite + crx({ manifest })
├── tsconfig.json             # strict, types: chrome + vite/client
├── package.json
├── src/
│   ├── background/
│   │   └── index.ts          # service worker: приём SUMMARIZE, вызов LLM (Этап 1)
│   ├── content/
│   │   └── index.ts          # content script: извлечение текста/выделения, PDF (Этап 4)
│   ├── popup/
│   │   ├── index.html        # entry popup
│   │   └── popup.ts          # логика popup (голый TS)
│   └── lib/                  # общий код всех контекстов
│       ├── messages.ts       # типизированный контракт сообщений (bg/content/popup)
│       └── settings.ts       # BYOK-настройки в chrome.storage
├── docs/                     # PLAN / TECHNICAL_PLAN / POST_MVP_PLAN
└── .claude/skills/browser-extension-dev/SKILL.md
```

**Почему `src/lib/messages.ts` отдельным модулем.** Три контекста расширения
(content / background / popup) исполняются в изолированных мирах и общаются
**только** через message-passing. Единый типизированный контракт сообщений
устраняет рассинхрон строковых литералов между контекстами.

### 3.1 Поток данных (саммаризация страницы, Этап 1)

```
popup: клик → chrome.tabs.sendMessage(tabId, {EXTRACT})
   → content: извлекает текст → sendResponse({EXTRACT_RESULT, text, url, title})
popup: → chrome.runtime.sendMessage({SUMMARIZE, text})
   → background(service worker): loadSettings() → fetch(baseUrl + /chat/completions)
        [BYOK-ключ из chrome.storage, retry/обработка ошибок]
   → sendResponse({SUMMARIZE_RESULT, ok, summary|error})
popup: рендер сводки/ошибки
```

## 4. Жизненный цикл service worker (MV3) — критично

Service worker в MV3 **эфемерный**: браузер выгружает его при простое и будит на
событие. Отсюда правила (детали — в `SKILL.md`):

- **Слушатели регистрируются синхронно на верхнем уровне** модуля (не внутри
  async-колбэков) — иначе после пробуждения worker их не привяжет.
- **Никакого важного состояния в module-level переменных** — не переживёт
  выгрузку. Настройки/кэш — в `chrome.storage`.
- В `onMessage` при async-ответе — **`return true`**, чтобы канал `sendResponse`
  не закрылся раньше времени.

## 5. Popup на голом TS, без React — обоснование

Popup MVP — одна кнопка, область результата и небольшая форма настроек. React
добавил бы вес бандла и сложность сборки при около-нулевой выгоде. Голый TS +
DOM API достаточно и полезнее для тренировки базовых API. Если side panel
(Этап 6) вырастет в богатый UI — пересмотреть тогда, точечно, не раньше.

## 6. LLM-доступ (Этап 1–2)

- Прямой `fetch` из service worker к `POST {baseUrl}/chat/completions`
  (OpenAI-совместимый контракт).
- Заголовок `Authorization: Bearer <apiKey>`; **ключ читается из
  `chrome.storage.local`** в момент вызова, не кэшируется в модуле, не логируется.
- Обработка: таймаут через `AbortController`, разбор ошибок (401 → «проверьте
  ключ», 429 → «лимит»), усечение входного текста под потолок токенов (Этап 3).
- CORS: top-level `fetch` из origin расширения к пользовательскому endpoint
  обычно проходит; при необходимости конкретного хоста — добавить в
  `host_permissions` (сейчас пусто, полагаемся на `activeTab`).

## 7. PDF (Этап 4)

- Зависимость `pdfjs-dist`. Content script определяет PDF-контекст: либо URL
  оканчивается на `.pdf`/`content-type: application/pdf`, либо страница —
  встроенный PDF-вьюер браузера.
- Извлечение: загрузить документ через `pdfjs`, пройти страницы, собрать
  `textContent` в единый текст, дальше — обычный поток саммаризации.
- Воркер `pdf.js` подключается как отдельный ассет сборки (Vite `?url`/
  `web_accessible_resources`); детали — при реализации Этапа 4.

## 8. Разбивка по этапам (deliverable + критерий приёмки)

### Этап 0 — Bootstrap ✅ (готово)

- **Действия:** `package.json`, Vite + `@crxjs`, `tsconfig` (strict),
  `manifest.config.ts` (MV3), заглушки `background/index.ts`,
  `content/index.ts`, `popup/index.html` + `popup.ts`, `lib/messages.ts`,
  `lib/settings.ts`.
- **Deliverable:** каркас без бизнес-логики; типизированный контракт сообщений.
- **Критерий:** `npm install` и `npm run build` (tsc + vite) зелёные;
  `dist/manifest.json` валиден; расширение грузится в `chrome://extensions`
  (Load unpacked → `dist/`). **Выполнено.**

### Этап 1 — Саммаризация всей страницы

- **Действия:** извлечение читаемого текста в content script (замена наивного
  `innerText` на очистку от nav/ads); LLM-вызов в service worker; рендер сводки
  в popup; сквозной message-passing.
- **Deliverable:** кнопка «Summarize page» выдаёт реальную сводку.
- **Критерий:** на живой статье popup показывает осмысленную сводку; юнит-тесты
  (Vitest) на извлечение текста и сборку запроса зелёные.

### Этап 2 — Настройки (BYOK)

- **Действия:** форма настроек (base URL, ключ, модель) в popup; сохранение/
  загрузка через `chrome.storage.local` (`lib/settings.ts`); понятное сообщение,
  если ключ не задан.
- **Deliverable:** пользователь вводит свой ключ, он переживает перезапуск.
- **Критерий:** без ключа — дружелюбная подсказка, не падение; ключ не
  появляется в логах/сборке; тесты на load/save settings.

### Этап 3 — Устойчивость и UX

- **Действия:** обработка ошибок сети/401/429; усечение длинного текста под
  лимит токенов; индикатор загрузки; отмена запроса (`AbortController`).
- **Deliverable:** предсказуемое поведение на ошибках и длинных страницах.
- **Критерий:** искусственные 401/429/таймаут дают понятные сообщения;
  очень длинная страница не роняет запрос.

### Этап 4 — Поддержка PDF

- **Действия:** `pdfjs-dist`; детект PDF-контекста; извлечение текста из PDF, в
  т.ч. встроенного вьюера; подключение воркера pdf.js в сборке.
- **Deliverable:** саммаризация PDF, открытого в браузере.
- **Критерий:** на тестовом PDF извлекается текст и строится сводка; не-PDF
  страницы не задеты.

### Этап 5 — Саммаризация выделенного

- **Действия:** пункт контекстного меню (`chrome.contextMenus`) «Summarize
  selection»; передача выделения в поток саммаризации; результат в popup/side
  panel.
- **Deliverable:** правый клик по выделению → сводка фрагмента.
- **Критерий:** выделение корректно доходит до LLM; пустое выделение —
  понятное сообщение.

### Этап 6 — Side panel (конец MVP)

- **Действия:** Chrome Side Panel API; панель как альтернатива popup для длинных
  сводок; переиспользование той же логики саммаризации.
- **Deliverable:** сводка читается в боковой панели.
- **Критерий:** панель открывается, показывает сводку, popup продолжает
  работать; общий код не продублирован.

## 9. Docker Compose — почему НЕ нужен

Расширение **полностью клиентское**: нет сервера, БД, очередей — оркестрировать
контейнерами нечего. Docker Compose в проект **не добавляется**.

Единственный теоретический кандидат — сервис `ollama` для локального теста
саммаризации без облачного ключа. Он отклонён на MVP: (1) архитектура и так
BYOK — достаточно указать в настройках локальный OpenAI-совместимый endpoint
(тот же Ollama/LM Studio на `http://localhost:11434/v1`), запущенный вне
контейнера; (2) тянуть Docker ради одного опционального dev-провайдера —
несоразмерно узкому скоупу. Если позже понадобится детерминированный локальный
фейк-endpoint для интеграционных тестов — вернуться к вопросу точечно (кандидат
в `POST_MVP_PLAN.md`), а не заводить compose «на всякий случай».
